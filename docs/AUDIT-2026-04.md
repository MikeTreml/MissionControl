# MC audit ŌĆö 2026-04 (post journal-rightbar merge)

A categorized review of every renderer surface, persisted shape, IPC
channel, and error path. Goal: surface what feels wrong, missing,
unneeded, or nice-to-have. Reservations called out so we can tell
"validated finding" from "speculation worth raising."

Companion to [COMP-RESEARCH.md](COMP-RESEARCH.md). Where comp research
asks "what should we do," this audit asks "what's already wrong."

## Methodology

- Walked every page / component / hook (~25 files) and every IPC
  channel (45 methods).
- Cross-checked persisted shapes (`src/shared/models.ts`) against
  read sites in renderer.
- Catalogued silent failures, hardcoded values, missing UI states.
- A subagent did the raw data gathering; I reviewed and corrected
  several false positives. Anything below is verified.

## Caveats

- The codebase is mid-stride on the journal-rightbar merge; some
  recently-added IPCs (`runStatus`, `runListPending`) are deliberately
  un-consumed in the renderer awaiting migration. Those aren't
  "unused" ŌĆö they're "next."
- The audit looks at the *current branch* state. There may be
  improvements on other branches I haven't pulled in.
- Severity calls reflect my opinion; push back if any feel off.

## 1 ┬Ę Critical findings (block release)

### C1. Silent failures across 7 user-mutation paths

**What**: 7 places in the renderer log to `console.error` and don't
surface the failure to the user ŌĆö the form just stops, the action
silently no-ops.

**Sites**:
- [TaskDetail.tsx:189](src/renderer/src/pages/TaskDetail.tsx) ŌĆö Archive toggle
- [TaskDetail.tsx:226](src/renderer/src/pages/TaskDetail.tsx) ŌĆö Delete
- [TaskDetail.tsx:414](src/renderer/src/pages/TaskDetail.tsx) ŌĆö BlockerField save
- [AskUserCard.tsx:73](src/renderer/src/components/AskUserCard.tsx) ŌĆö cancelAsk
- [EditTaskForm.tsx:76](src/renderer/src/components/EditTaskForm.tsx) ŌĆö saveTask
- [AddProjectForm.tsx:134](src/renderer/src/components/AddProjectForm.tsx) ŌĆö submit
- [AddProjectForm.tsx:158](src/renderer/src/components/AddProjectForm.tsx) ŌĆö delete

**Why critical**: a user clicking Archive/Save/Delete and seeing
nothing happen has no way to know whether they need to retry, check
DevTools, or assume success. `Toaster.tsx` exists; it's just not
wired here.

**Fix**: covered by **#17** (already on the list ŌĆö moving up in
priority).

### C2. No loading states across pages

**What**: every page either renders blank during fetch or pops
data in unannounced. No skeletons, no spinners, no "LoadingŌĆ”"
stubs anywhere except inline button busy states.

**Why critical**: on first launch the user sees an empty Board for
~200ms. On a slow disk, longer. Looks broken.

**Fix**: covered by **#18** (already on list).

### C3. Demo-mode fallback is silent and reversible

**What**: every hook (`useTasks`, `useProjects`, `useTask`) falls
back to `mockTasks` / `mockProjects` if `window.mc` is missing OR if
the real list returns empty OR if any IPC throws. There's no
indicator that the user is *now* in demo mode mid-session.

**Why critical**: a real failure mode ŌĆö IPC briefly errors mid-session,
the UI swaps to demo data, the user sees their tasks "vanish" and
"new tasks appear." Confusing and unrecoverable without restart.

**Reservation**: my read of the code suggests the fall-through
happens on *fetch failure*, not *every empty result*. Need to verify
whether a real-empty Board (no tasks) shows demo or empty.

**Fix**: distinguish "preload missing" (reload the app) from "store
returned empty" (show empty state, **not** demo data) from "fetch
errored" (show error banner). At minimum: the "Demo" pill in the
Topbar should turn red and persist after a real failure, not silently
go away.

### C4. Renderer has zero smoke tests

**What**: `npm run smoke` covers main-process modules. Renderer pages
and hooks have no equivalent. UI integration bugs surface only when
the user clicks them.

**Why critical**: every UI commit could break a flow we don't
re-verify. The Playwright skeleton (`scripts/verify-ui.mjs`) covers
3 flows; we have ~12 user flows worth covering.

**Fix**: covered by **#24** (extend verify-ui.mjs), but should be
*ongoing* ŌĆö every new UI flow ships with at least one assertion.

## 2 ┬Ę Important findings (worth fixing soon)

### I1. Truly unused IPCs (5 methods)

After cross-checking, these IPCs have *zero* renderer call sites and
no plausible near-term consumer:

- `appendTaskEvent` ([global.d.ts:70](src/renderer/src/global.d.ts:70))
- `readTaskFile` ([:78](src/renderer/src/global.d.ts:78))
- `listTaskFileCycles` ([:82](src/renderer/src/global.d.ts:82))
- `appendTaskStatus` ([:83](src/renderer/src/global.d.ts:83))
- `getProject` ([:89](src/renderer/src/global.d.ts:89)) ŌĆö superseded by `listProjects` + renderer-side filter

**Reservation**: `appendTaskEvent` and `appendTaskStatus` *might* be
intentionally exposed for a future renderer-side use case (e.g.,
manually annotating a task). If so, mark them with `// PROPOSED:`
comments instead of removing.

**Fix**: prune the unambiguously dead ones (`getProject`,
`listTaskFileCycles`); leave a marker on the speculative ones.

### I2. `runStatus` IPC is exposed but unused (deliberate)

**What**: `window.mc.runStatus(taskId)` was added in the
journal-rightbar work as the SDK-authoritative state query, but
nothing in the renderer calls it yet.

**Why important**: today PhaseChipStrip's "current" chip is derived
from journal events. SDK has the authoritative answer in `runs:status`.
Migration is **#20** on the list.

**Fix**: covered by #20.

### I3. `BOARD_STAGES` hardcoded enum violates rule #4

**What**: `["Draft", "Active", "Attention", "Failed", "Complete"]`
hardcoded at [Board.tsx:6](src/renderer/src/components/Board.tsx:6).

**Why important**: workflows are supposed to drive lanes per
UI-DESIGN.md ┬¦5. Changing the workflow shouldn't require a code
change to the Board.

**Reservation**: workflow-driven lanes is a bigger redesign (#12 +
#27). The current hardcoded set might be the right *fallback* even
post-redesign ŌĆö workflows that don't declare their own lanes get
this set.

**Fix**: covered by #12; reservation noted.

### I4. `Task.updatedAt` is read for idle detection on ProjectDetail
only

**What**: my earlier read of ProjectDetail.tsx showed `idleHours()`
using `Task.updatedAt`. The audit agent claimed it was unused ŌĆö I
spot-verified and the agent was wrong; `updatedAt` IS used in
ProjectDetail's `StuckTable` and `computeStats`.

**Why important** (still): it's only used for *aggregation*. There's
no per-task "idle since X" indicator on TaskCard. A task can sit
idle for days and the Board shows nothing different than fresh.

**Fix**: small UI add ŌĆö show "idle 3h" / "idle 2d" muted text on
TaskCard for non-running, non-done tasks. New task surfaced below.

### I5. Hardcoded library item kinds and domain groups

**What**:
- `["agent", "skill", "workflow", "example"]` at
  `src/renderer/src/pages/Library/FilterBar.tsx:37` and
  `src/renderer/src/pages/Library/DetailPanel.tsx:5`
- `["", "business", "science", "social-sciences-humanities"]` as
  const at `src/renderer/src/pages/Library/DetailPanel.tsx:17`
- `["AGENT.md", "SKILL.md", "workflow.js"]` at
  `src/main/run-cost-tracker.ts:71`
- Domain list duplicated at `run-cost-tracker.ts:72`

**Why important**: domain list will grow; the duplication will
silently drift. Library kinds *might* extend (e.g., "process",
"prompt-template").

**Reservation**: these are real library structure constants. They're
declared in code because the library schema declares them. If we
treat the library as the source of truth, the constants should be
loaded from `library/_index.json` metadata, not hardcoded.

**Fix**: surface a new task ŌĆö derive these from the library index.

### I6. ChangeWorkflowModal callsites suggest `saveWorkflowRunTemplate`,
not its actual purpose

**What**: the audit agent inferred from naming that
ChangeWorkflowModal calls `saveWorkflowRunTemplate`. I haven't
verified.

**Reservation**: needs a quick read of ChangeWorkflowModal to confirm
it's writing `RUN_CONFIG.json` and not also saving as a template.
If both, the UX may be confusing.

**Fix**: triage task ŌĆö verify and document.

### I7. ProjectDetail's StuckTable references a stale lane vocabulary

**What**: the StuckTable filter uses `t.lane === "Waiting"` and
`t.roleLabel === "Waiting"`. This is current; the post-Phase-10
lane axis is `Idle | Running | Waiting | Done | Failed`. So the code
*is* correct.

**But**: a comment near `LaneBars` says it explicitly avoids the
"old kanban-lane vocabulary." Two layers using the same word "lane"
for different things is confusing.

**Fix**: small rename pass ŌĆö the kanban stage axis (Draft / Active /
Attention / Failed / Complete) and the run-state axis (Idle / Running
/ Waiting / Done / Failed) should have distinct names. Surface as
new task.

## 5 ┬Ę Nice-to-haves surfaced

### N1. Per-task idle indicator on TaskCard

Already noted in I4. "idle 3h" muted line on cards for non-running,
non-done tasks. Cheap.

### N2. "Demo mode" indicator persistence

Today the Topbar shows a "Demo" pill when `isDemo`. Make it
*sticky* if the user was in real mode and got bumped to demo by an
IPC failure ŌĆö turns red, requires explicit dismissal or app restart.

### N3. `appendTaskStatus` for user-added notes

Today STATUS.md is append-only by main process during runs. A "Add
note" button on TaskDetail that calls `window.mc.appendTaskStatus`
would convert the unused IPC into a feature: humans annotating
their own runs.

**Reservation**: small concept; might be the kind of thing nobody
uses. Worth trying because it's basically free given the IPC
already exists.

### N4. `listTaskFileCycles` ŌĆö show cycle history per task

Today TaskDetail shows the latest version of task files. The IPC
`listTaskFileCycles` returns the cycle numbers a file went through.
A "Cycle 1, Cycle 2, Cycle 3 (current)" picker would let users see
how a file evolved across reviewer loopbacks.

**Reservation**: niche. Useful when reviewing a campaign or a
multi-cycle run; useless for one-shot. Not urgent.

### #52. Document Topbar status dot color semantics

## 8 ┬Ę Reservations worth raising

Things I noticed that I'm *not sure about*. Surface for review:

### R1. The dashboard is a thin composition; could be deleted

`Dashboard.tsx` is just `<Topbar /><Board /><SelectedTaskPanel />`.
The Topbar already routes; the Board is the content. Could the
Board page directly be the dashboard, eliminating the wrapper?

**Why I'm uncertain**: SelectedTaskPanel exists for a reason
(probably keyboard nav?). Need to verify.

### R2. RunMetricsCard / aggregateProjectRunMetrics ŌĆö overlap?

Per-task metrics card on TaskDetail + per-project rollup on
ProjectDetail. They presumably read the same `artifacts/*.metrics.json`.
Worth confirming they agree.

### R3. The "Metrics" page has its own cross-project rollup

Three places now show metrics: TaskDetail, ProjectDetail, Metrics
page. Consistent presentation? Same source data? Worth a small UX
audit.

### R4. Settings page only has a "Global" subtab

The implication is there will be more subtabs (per-project? per-
workflow?). If we know they'll exist, scaffold them now (empty
panels with "Coming soon"). If not, drop the subtab UI and just
show settings flat.

### R5. `BlockerField` in Task Detail ŌĆö does this make sense post-cleanup?

Tasks have a `blocker` string field. It feeds into "Waiting" lane
derivation. With the journal-rightbar approval flow, blockers are
surfaced through `BREAKPOINT_OPENED` events instead. Is the manual
`blocker` field still useful, or vestigial?

### R6. The mockup mentioned "compare workflows" ŌĆö we have nothing

User mentioned this idea (#30 worktree toggle) but no UI surface
exists for actually *comparing* two runs side-by-side. The audit
didn't surface a comparison primitive anywhere. Comp research item.
