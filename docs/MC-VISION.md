# Mission Control — Vision

> **Audience:** Michael. Note-to-self for orientation. Not marketing, not a
> handoff doc. Coding-assistant context lives in `CLAUDE.md` / `AGENTS.md`
> at the repo root — this file only repeats what you'd want to re-read on
> a fresh sit-down to remember why MC exists and where it's going.

---

## 1. What MC is, in one sentence

Mission Control is a single-operator desktop app where Michael drives AI
coding agents across many projects: the UI + state + project bookkeeping
layer on top of a rented orchestration runtime (`@a5c-ai/babysitter-sdk`)
and a rented agent runtime (`@mariozechner/pi-coding-agent`, with
`oh-my-pi` as the preferred direction).

---

## 2. Who it's for

- **Michael, solo.** That's the whole user list.
- Not a product to ship to other people. No tenancy, no auth, no sharing.
- The desktop app shape exists because Michael wants a single visible
  surface for many concurrent runs across many repos — not because
  Electron is strategically required.

If anything ever changes here, this section is the canary — broaden the
audience and most of §8 ("Not in scope") needs revisiting at the same time.

---

## 3. Ownership boundaries

MC keeps three things and rents the rest.

| Layer | Owned by | Why |
|---|---|---|
| Orchestration runtime — runs, journals, effects, replay, hooks | **Babysitter SDK** | Deterministic replay + event-sourced runs is hard. Don't reinvent. |
| Agent runtime — model dispatch, sessions, tool execution | **Pi / OMP** | Same reasoning. Pi's session model + tool surface is mature; we use it. |
| UI, project bookkeeping, per-task state, IPC plumbing | **MC** | This is the actual product surface — what Michael sees and clicks. |

Practical rule: if the question is "who handles X?" and X is "running
a process," "tracking effects," "loading a model," "executing a tool" —
the answer is one of the rented layers. MC's job is to **glue** those
layers to a usable interface and to persist the parts (project + task
metadata) that only matter to the operator.

## 4. Two start paths

When you click Start on a task, exactly one of two things happens:

**Curated path** (preferred when you know what you want):
```
Library → pick workflow → RUN_CONFIG.libraryWorkflow.diskPath
  → RunManager.startCuratedWorkflow
    → babysitter run:create --entry <workflow.js>#process
    → MC drives run:iterate in a loop
      → for each pending agent effect: dispatch via claude headless
      → for each pending shell effect: spawn the command
      → task:post each result
    → run reaches RUN_COMPLETED or RUN_FAILED
```

**Auto-gen fallback** (when the work is ad-hoc):
```
No curated workflow set
  → pi.session.prompt('/babysit <brief>')
  → babysitter generates a process.js per task
  → drives the run from inside the pi session
```

The curated path is where quality gates live. The auto-gen path is for
"I just want pi to figure something out." Both end in the same on-disk
shape (`<workspace>/.a5c/runs/<runId>/`), so MC's UI is one set of views
over both.

## 5. The library

`library/` is the catalog of reusable building blocks. Four item kinds,
fixed locations:

- **process** → `library/specializations/<spec>/<name>.js` — runnable
  babysitter workflow.
- **skill** → `library/specializations/<spec>/skills/<name>/SKILL.md` —
  prose injected into an agent's system prompt at runtime via the SDK's
  skill resolver.
- **agent** → `library/specializations/<spec>/agents/<name>/AGENT.md` —
  catalog persona. Browseable, discoverable. **Not loaded at runtime
  today** (the SDK never reads AGENT.md bodies into prompts).
- **methodology** → `library/methodologies/<name>/` — patterns + agents +
  skills bundled around a development methodology.

Two principles govern what goes in:

1. **Curate, don't filter.** Keep the catalog tight. A dynamic-selection
   layer that picks "what's helpful" from thousands of items at runtime
   is unreliable; pre-pruning is deterministic. Only add items you
   actually use or are actively evaluating.
2. **Inline before catalog.** When persona text only applies to one
   workflow, write it inline in the task's `agent.prompt.role`. Promote
   to a SKILL.md only when reuse is real and demonstrated.

Indexes (`_index.workflow.json` etc.) are MC-side build artifacts —
regenerated by `npm run build-library-index`. The upstream
process-library checkout doesn't carry indexes; that's fine, MC doesn't
need them there.

## 6. The development workflow MC uses on itself

**Methodology: evolutionary** — incremental change, low ceremony,
matches the solo trunk-based push-to-main reality. Locked in CLAUDE.md.

**Loop:** `.a5c/processes/continue-mission-control-with-quality.js`

```
analyze repo state → human-approve breakpoint → implement
  → TYPECHECK GATE (npm run typecheck — exit 0 required)
  → AGENT-SCORED REVIEW (5-dimension rubric, target 80, max 5 attempts)
  → final verification (smoke + tsc) only on convergence
  → summary with follow-ups
```

**Quality knobs (per-run inputs):**

| Input | Default | Override when |
|---|---|---|
| `targetQuality` | 80 | Higher (85–90) for changes touching persisted contracts or UI; lower (70) for throwaway exploration. |
| `maxIterations` | 5 | Lower (2–3) for tiny tweaks where you'd rather bail than grind. |

**On non-convergence the diff is kept in the working tree** — no
auto-revert. Manual functional review takes over from there.

**Pre-commit gate:** `npm run doctor` (typecheck + smoke + workflow
guards + build + verify-ui). The quality-gated workflow's in-loop
typecheck overlaps the first gate; doctor is the additional pass before
push.

The original `continue-mission-control.js` (no quality gates) stays as
the lighter alternative when the quality loop is too much ceremony.

## 7. Strategic direction

What we're walking toward, in priority order:

1. **OMP as orchestrator.** Use OMP's loop function plus
   `babysitter harness:forever` / `yolo` for ongoing operator work
   (cron-style chores, watching tasks, periodic checks). Doesn't need
   workflows — set rules, let it run. Currently a "want" not a "need";
   Codex + Claude are fine in the meantime.
2. **Per-task model micromanagement.** The SDK already supports
   `execution.model` per task. Surface this in MC where it pays off —
   cheaper models on simple phases, stronger models on quality gates.
3. **Library inputs UI.** The pipe from `RUN_CONFIG.libraryWorkflow.inputs`
   to the workflow already works. The Library Run modal needs a per-
   workflow form that reads `@inputs` JSDoc and renders the right fields.
   `targetQuality` is the first to surface.
4. **Tight, curated library** — keep pruning. Items earn their place
   through actual reuse, not speculation.

What we're explicitly not chasing — see §8.

## 8. Deliberately NOT in scope

Things that have come up and been ruled out, kept here so the same
question doesn't get re-asked from zero:

- **Multi-user / multi-tenancy.** §2 audience is the contract. No auth,
  no per-user state, no permission model.
- **Cloud sync, sharing, public catalog/marketplace.** All state stays
  on disk under `<userData>/` and `.a5c/`. Inspect-friendly,
  git-friendly when you want.
- **Native modules** (anything that compiles to a `.dll`/`.so`/`.dylib`).
  Breaks Electron's bundled binary. Pi pinning is already a minefield
  per CLAUDE.md "Dependencies".
- **Big state-management libs** (Redux, Zustand, MobX).
  `useState` + `data-bus.ts` is sufficient for the scale of UI.
- **UI component libraries** (Material, Chakra, Radix at app level).
  Custom CSS with the token palette is the convention.
- **Test frameworks needing a bundler** (Jest, Mocha). The `.smoke.ts`
  pattern runs standalone via `node --experimental-strip-types`.
- **Agent-materialization runtime infrastructure.** Wiring AGENT.md →
  workspace `.claude/agents/` per harness is doable but more code than
  the curated-catalog approach justifies. The SDK's existing skill-
  injection path covers the same use case for items that genuinely
  need runtime presence.
- **SQLite / heavy persistence layers.** File-first JSON + markdown is
  the rule. Inspectable, git-able, debuggable without tooling.
- **Hardcoded labels / model ids / project prefixes.** If a value comes
  from a file at runtime, read it. Don't bake.
- **Reintroducing the `harness:create-run --process` curated argv.**
  Empirically broken in SDK 0.0.187 (Phase-2 path-undefined crash when
  Phase 1 is skipped). The `run:create --entry` + caller-driven loop
  replaced it on 2026-05-06.
